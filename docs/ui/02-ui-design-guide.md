# UI 设计指南

本文档面向 UI 设计师，用来把训练模块第一阶段的用户流程、页面功能、状态和指标转成可设计的界面方案。设计时不需要理解后台实现，只需要围绕用户在训练前、训练中、训练后的决策和反馈来组织页面。

相关产品方案见 `docs/plans/[done]-training-redesign.md`。

## 设计目标

第一阶段只做“计划训练”的完整闭环：

```text
训练首页
  -> 动作库
  -> 训练计划库
  -> 计划编辑
  -> 训练执行
  -> 训练摘要
  -> 保存或丢弃结果
  -> 返回训练首页
```

核心体验目标：

- 用户进入训练页后，能清楚知道当前是否有可开始的训练计划。
- 用户能用动作库和计划编辑器搭建训练内容。
- 用户开始训练后，界面优先显示当前阶段倒计时、动作、Set / Rep 和设备反馈。
- 用户训练结束后，能快速判断本次训练完成情况，并选择保存或不保存。

第一阶段不设计：

- 自由训练。
- 新记录列表、记录详情和历史趋势。
- 力竭信号。
- 训练中跳过阶段。
- 训练中临时修改计划。
- App 被系统终止后的恢复流程。

## 核心概念

### 动作

动作是可复用的训练模板。例如：

```text
动作名称：左手悬挂
动作类型：timed reps
次数：12 reps
每次锻炼：12s
组内休息：4s
```

第一阶段只支持 `timed reps`：

- 一个 rep = work n 秒。
- rep 之间可以有 rest n 秒。
- 最后一个 rep 后没有 rest。

### 训练计划

训练计划是一组按顺序执行的步骤。步骤可以是：

- 动作组。
- 计划间隔 interval。

### 动作组

动作组是训练计划内部的一段循环结构。它可以只有一个动作，也可以由多个动作组合而成。

例子：

```text
动作组：左右手悬挂
  interval：准备 10s
  action：左手悬挂
  interval：换手 15s
  action：右手悬挂
组间休息：60s
循环次数：3
```

动作组规则：

- 每个动作组至少要有 1 个有效动作。
- `interval` 可以放在开头、中间或结尾。
- 允许连续多个 `interval`，并且它们要作为独立阶段显示。
- 组间休息只发生在 cycle 之间，最后一轮之后没有组间休息。

### 测量模式

开始训练时根据设备状态确定模式：

- `forceDevice`：已连接拉力计。
- `timerOnly`：未连接拉力计。

开始后模式固定：

- `timerOnly` 训练中不切换成 `forceDevice`。
- `forceDevice` 断连后进入意外暂停，不降级为 `timerOnly`。

## 全局导航结构

训练模块第一阶段建议围绕一个主 Tab 页面展开：

```text
训练 Tab
  训练首页
    动作库 sheet / page
      动作编辑 sheet / page
    训练计划库 sheet / page
      计划编辑 sheet / page
    训练执行 full screen
      训练摘要 sheet / page
```

设计师需要产出的主要页面：

- 训练首页。
- 动作库。
- 动作编辑。
- 计划库。
- 计划编辑。
- 训练执行。
- 暂停状态。
- 意外暂停 / 设备断连状态。
- 训练摘要。
- 删除 / 保存 / 丢弃等确认弹窗。

## 用户流程

### 流程 1：选择计划并训练

```text
进入训练首页
  -> 选择训练计划
  -> 可选：连接设备
  -> 开始训练
  -> 实时执行计划阶段
  -> 训练自然完成，或用户手动结束
  -> 查看训练摘要
  -> 保存训练结果
  -> 返回训练首页
```

关键设计点：

- 连接设备不是开始训练的硬性要求。
- 未连接设备时，点击开始训练不弹二次确认，直接进入倒计时训练。
- 手动结束训练需要确认。
- 训练完成后，摘要是“未保存”状态，用户必须明确选择保存或不保存。

### 流程 2：自定义训练计划

```text
进入训练首页
  -> 打开计划库
  -> 新建或编辑计划
  -> 添加动作组
  -> 在动作组内添加 interval / action
  -> 设置组间休息
  -> 设置循环次数
  -> 保存计划
  -> 返回计划库或训练首页
```

关键设计点：

- 计划编辑过程允许临时无效，例如还没有动作。
- 保存时必须有效。
- 新建计划保存后，自动成为当前计划。
- 编辑当前计划保存后，当前计划保持不变。
- 编辑非当前计划保存后，不自动切换当前计划。

### 流程 3：管理动作库

```text
进入动作库
  -> 新建动作
  -> 编辑动作
  -> 删除动作
```

关键设计点：

- 动作名称必须唯一。
- 修改一个已被计划引用的动作，会影响相关训练计划。
- 保存已被引用动作前，需要提醒并列出受影响计划。
- 操作选项：
  - 保存并更新训练计划。
  - 保存为新动作。
  - 取消。
- 删除被引用动作时，允许删除，但要列出受影响计划；删除后相关计划变为无效。

## 页面设计说明

### 1. 训练首页

页面目的：让用户快速判断能不能开始训练，以及下一步该做什么。

首页主要模块：

- 当前计划区域。
- 设备状态区域。
- 训练库入口：计划库、动作库。
- 底部主操作按钮。

状态 A：没有训练计划

```text
标题：暂无训练计划
主操作：新建计划
次入口：动作库
```

状态 B：有计划，但未选择当前计划

```text
标题：未选择训练计划
主操作：选择计划
次操作：新建计划
```

状态 C：当前计划有效

```text
显示：
  计划名称
  动作数量
  预计训练时间
  设备状态

主操作：
  开始训练

次入口：
  编辑计划
  切换计划
  计划库
  动作库
```

状态 D：当前计划无效

```text
显示：
  计划名称
  无效原因

主操作：
  修复计划

次操作：
  选择其他计划

开始训练：
  不可用
```

设备状态建议：

- 已连接：显示连接成功状态，可以开始 `forceDevice` 训练。
- 未连接：显示未连接状态，仍可开始 `timerOnly` 训练。
- 设备操作：连接 / 断开。

### 2. 动作库

页面目的：管理可复用动作。

列表项建议显示：

```text
动作名称
12 reps · work 12s · rest 4s
```

页面操作：

- 新增动作。
- 点击动作进入编辑。
- 删除动作。

空状态：

```text
暂无动作
新建动作后可在计划中使用
主操作：新增动作
```

删除确认：

```text
删除“左手悬挂”？
```

如果动作被计划引用：

```text
删除“左手悬挂”会让以下计划失效：
最大指力、耐力训练

操作：
  删除动作
  取消
```

### 3. 动作编辑

页面目的：创建或修改一个 timed reps 动作。

字段：

```text
动作名称
次数 reps
每次锻炼 work seconds
组内休息 rest seconds
```

字段范围：

```text
次数：1...200
work：1...600 秒
rest：0...600 秒
```

保存校验：

- 名称不能为空。
- 名称不能与已有动作重复。
- 数值必须在范围内。

当动作已被计划引用时，需要显示影响提示：

```text
保存并更新会影响：
最大指力、左右手训练
```

操作：

- 保存并更新。
- 保存为新动作。
- 取消。

注意：

- “保存为新动作”需要使用新的动作名称。
- “保存为新动作”不会修改已有训练计划。

### 4. 训练计划库

页面目的：查看、选择、编辑训练计划。

列表项建议显示：

```text
计划名称
3 个动作 · 预计 18:40
```

无效计划也要显示，但需要有清晰的异常提示：

```text
计划名称
缺失动作：左手悬挂
状态：无效
```

页面操作：

- 新增计划。
- 选择当前计划。
- 编辑计划。
- 删除计划。

当前计划建议有明显标记，例如 checkmark 或“当前”标签。

删除当前计划后：

- 清空当前选择。
- 不自动切换到其他计划。

### 5. 计划编辑

页面目的：让用户组装训练计划。

计划级字段：

```text
计划名称
步骤列表
```

计划级操作：

- 新增动作组。
- 新增计划间隔。
- 调整步骤顺序。
- 删除步骤。

动作组卡片 / 区块需要显示：

```text
动作组名称，可选
组内步骤列表
组间休息
循环次数
```

动作组内步骤可以是：

```text
interval：名称，可选 + 时长
action：动作引用
```

动作组内操作：

- 添加 interval。
- 添加动作。
- 内联新建动作。
- 替换动作。
- 调整步骤顺序。
- 删除步骤。

保存计划校验：

- 计划名称不能为空。
- 计划名称不能重复。
- 至少有 1 个动作组。
- 每个动作组至少有 1 个有效动作。
- 至少有 1 个有效动作引用。
- 数值必须在范围内。

建议设计重点：

- 计划编辑器要让“层级”足够清楚：计划步骤 -> 动作组 -> 组内步骤。
- interval 和 action 需要有明显不同的视觉类型。
- cycle 和 group rest 是动作组级设置，不是单个 action 的设置。

### 6. 训练执行页

页面目的：训练时只保留必要信息，让用户一眼知道现在该做什么。

主信息优先级：

```text
当前阶段
当前阶段剩余时间
当前动作名称
Set / Rep
动作组进度
计划总进度
```

阶段类型：

- work：锻炼。
- rep rest：组内休息。
- interval：间隔。
- group rest：组间休息。
- paused：暂停。
- resume countdown：准备恢复。

训练执行页操作：

- 暂停 / 恢复。
- 结束。

手动结束确认：

```text
结束本次训练？
  结束并查看摘要
  继续训练
```

#### forceDevice 模式

显示：

- 当前力量读数。
- 阶段曲线。
- Y 轴当前最大值可不显性展示，但设计需容纳动态扩展。

曲线规则：

- 每个非暂停阶段开始时，重置曲线视窗。
- X 轴最大值 = 当前阶段时长。
- Y 轴初始最大值 = 10kg。
- 如果当前读数超过 Y 轴最大值的 70%，Y 轴最大值增加 10kg。
- work 阶段曲线视觉更突出。
- rest / interval / group rest 曲线视觉弱化。
- paused 不绘制曲线。

#### timerOnly 模式

显示：

- 倒计时画面。
- 当前阶段。
- 当前动作 / Set / Rep。

不显示：

- 当前力量。
- 曲线。
- 最大力量。

#### 暂停状态

主动暂停和意外暂停都进入暂停画面。

暂停画面显示：

```text
已暂停
暂停时长或当前状态
恢复
结束
```

forceDevice 断连导致的意外暂停，需要显示设备断连状态：

```text
设备已断开
请重新连接后恢复训练

操作：
  重新连接
  恢复，未连接时不可用
  结束
```

恢复规则：

- 恢复前先 3 秒倒计时。
- 如果暂停发生在 work 阶段，恢复后重新开始该 work 阶段。
- 如果暂停发生在非 work 阶段，恢复后继续剩余时间。

### 7. 训练摘要

页面目的：让用户确认本次训练结果，并决定保存或丢弃。

摘要顶部状态：

```text
训练摘要
未保存
```

基础指标，所有模式都显示：

```text
预计时间
总时间
训练时间
暂停时长
包含动作数
```

时间口径：

- 总时间：从开始到结束的墙钟时间，包含暂停和恢复倒计时。
- 训练时间：不包含暂停和恢复倒计时的 active duration。
- 暂停时长：只显示汇总暂停时长，不拆分主动 / 意外暂停次数。

按动作统计：

```text
动作名称
完成组数
未完成组数
完成 reps
组间休息时间
```

forceDevice 额外显示：

```text
最大力量
rep 级 peak force
```

timerOnly 不显示：

```text
最大力量
rep peak
曲线
当前力量
```

摘要操作：

- 保存。
- 不保存。

不保存需要确认：

```text
不保存本次训练？
  不保存并返回
  取消
```

保存后：

- 返回训练首页。

不保存确认后：

- 丢弃本次训练。
- 返回训练首页。

## 信息展示建议

### 时长格式

建议统一：

```text
12s
1:20
18:40
```

长时间可使用：

```text
1h 12m
```

### 力量格式

建议统一：

```text
24.5 kg
```

小数位可根据实际读数策略统一为 1 位。

### 训练阶段文案

建议中文显示：

```text
锻炼
组内休息
间隔
组间休息
已暂停
准备恢复
```

### 无效状态文案

建议简短明确：

```text
缺失动作
动作参数无效
计划至少需要一个动作组
动作组至少需要一个动作
名称不能为空
名称不能重复
```

## 设计边界

设计时请避免把第一阶段没有实现的能力放进主流程：

- 不要设计训练中编辑计划。
- 不要设计训练中跳过当前阶段。
- 不要设计自由训练入口。
- 不要设计历史记录列表和详情。
- 不要设计力竭信号图标或指标。
- 不要设计 timerOnly 训练中途升级为 forceDevice 的流程。

可以预留未来扩展空间，但不要让第一阶段用户误以为这些能力已经可用。
